# Dead code

Backend codebases accumulate dead code faster than they accumulate live code. Anvil's stance: **dead code is debt with negative interest. Detect it. Remove it. Never let it sit "kept for reference."**

This file covers what counts as dead, how to detect it per language, the removal protocol, and the narrow cases where keeping it is defensible.

---

## What counts as dead

In decreasing order of confidence:

| Category | Confidence | Example |
| --- | --- | --- |
| **Unused exports** | High — tool can prove it within the module | `func ExportedHelper()` no one imports |
| **Unused internal helpers** | High — within a module, the call graph is closed | `func parseRow()` called nowhere |
| **Unreachable code** | High — control flow proves it | Code after `return` |
| **Empty interface implementations** | High when interface is mocked elsewhere or never extended | `Plugin` with one impl that no one wires up |
| **Quasi-dead** | Medium — called only from another dead path | A helper called only by an unused export |
| **Feature-flagged off forever** | Medium — flag exists but is always-off in config | `if cfg.EnableFoo { … }` and `cfg.EnableFoo` is hardcoded `false` |
| **Commented-out blocks** | High that it's dead, medium that it should be removed | `// old impl:\n// func foo() { … }` |
| **"Kept for reference" scraps** | Medium — feels safer to keep | `// TODO: maybe revive when we revisit X` |
| **TODO with no owner, no date** | Medium — silent decay | `// TODO: remove this hack` (from 2024) |
| **Deprecated public APIs past removal milestone** | High — anvil should remove on cue | `// Deprecated: removed in v3.0.0` and we're now on v3.2 |

**Not dead** (don't flag):
- Code behind a flag that's flipped per-environment (real feature flag)
- Test fixtures and golden files
- Examples in `examples/` or `_examples/` directories
- Public exports in a library where external callers exist
- Generated code clearly marked as such (`// Code generated by …`)
- Conditional compilation per platform (`//go:build linux`)

## Detection by language

Use language-native tools. Anvil does not invent its own dead-code analysis.

### Go

| Tool | Catches |
| --- | --- |
| `go vet ./...` | Basic unreachable code |
| `staticcheck ./...` (`U1000`, `SA9006`) | Unused vars, fields, types, funcs (private + public) |
| `unused` (now folded into staticcheck) | Unused symbols within module |
| `deadcode ./...` ([`golang.org/x/tools/cmd/deadcode`](https://pkg.go.dev/golang.org/x/tools/cmd/deadcode)) | Whole-program reachability from `main` |
| `go mod why -m <dep>` | Dependency reachability |
| `go-cleanarch` | Layered architecture violations |

For libraries, `deadcode` is less useful (no `main`); rely on `staticcheck U1000` + manual review of public surface.

```bash
deadcode -test ./...
staticcheck -checks U1000 ./...
```

### Rust

| Tool | Catches |
| --- | --- |
| `cargo check` + `#[warn(dead_code)]` (default warn) | Unused items |
| `cargo +nightly udeps` | Unused dependencies |
| `cargo machete` | Same, stable channel |
| `cargo-bloat` | Code size attribution (helps spot accidental large dead modules) |

```bash
cargo machete
cargo clippy -- -W dead_code
```

### TypeScript / Node

| Tool | Catches |
| --- | --- |
| [`knip`](https://github.com/webpro/knip) | Unused files, exports, deps, devDeps, types |
| `ts-prune` | Unused exports (knip's predecessor; knip is more thorough) |
| `eslint` with `no-unused-vars`, `import/no-unused-modules` | Per-file unused symbols |
| `depcheck` | Unused dependencies |

```bash
npx knip --reporter compact
```

Knip is the current gold standard — it understands monorepos, dynamic imports, framework conventions (Next.js, Nest, Astro), and config-driven entry points.

### Python

| Tool | Catches |
| --- | --- |
| `vulture` | Unused functions, classes, imports, variables |
| `pyflakes` / `ruff` (rules `F401`, `F841`) | Unused imports / locals |
| `coverage.py` reverse-reading | "Never executed" lines after a representative test run |

```bash
vulture src/ --min-confidence 80
ruff check --select F401,F841 .
```

`vulture` produces false positives — confidence rating helps. Combine with `coverage` for high-conviction targets.

### Java / Kotlin

| Tool | Catches |
| --- | --- |
| IntelliJ/Android Studio "unused declaration" inspection | Unused symbols (best UX) |
| `error-prone` `UnusedMethod`, `UnusedVariable` | Compile-time |
| `detekt` `UnusedPrivateMember` | Kotlin static analysis |

JetBrains IDEs are still the strongest dead-code finders in the JVM world.

### Multi-language repos

Run each language's tool separately. Don't try to unify under one tool. Aggregate the reports.

## The sweep protocol

`anvil reshape <target> --sweep` runs the dead-code campaign. **It never auto-deletes.** The protocol:

### Step 1 · Detect

Run the language-appropriate tool on the named target. Capture the report verbatim. Don't filter yet.

### Step 2 · Categorise

Bucket each finding into:

- **Safe** — unused exports + unused internal helpers, confidence high
- **Probably safe** — quasi-dead, commented-out blocks
- **Needs human** — feature-flagged paths, deprecated public APIs (need version check), TODO scraps
- **False positive** — generated code, reflection targets, plugin-registered handlers, struct tags read at runtime

### Step 3 · Plan

Emit `.anvil/sweep-report.md` (or to stdout if `--no-emit`):

```markdown
# Anvil sweep report
Target: internal/billing
Tool: staticcheck U1000 + deadcode
Date: 2026-05-22

## Safe to delete (12)

- internal/billing/parse.go:42 — `func parseRow(row []byte) Invoice` — no callers
- internal/billing/parse.go:91 — `func validateCheckSum(buf []byte) bool` — no callers
- internal/billing/store.go:18 — `var legacyTableName = "old_invoices"` — no readers
...

## Probably safe (4)

- internal/billing/util.go:7 — `func dec(s string) (int, error)` — only called by parseRow (above, dead)
- internal/billing/api.go:120-145 — commented-out POST /invoices/bulk handler
...

## Needs human (3)

- internal/billing/api.go:88 — `func handleLegacyEndpoint` (Deprecated: removed in v2.0; we're on v2.1)
- internal/api/feature.go:14 — `if cfg.EnableExperimentalBilling` — flag is hardcoded false in prod for 14 months
- internal/billing/TODO — `// TODO: revisit when we ship multi-currency` (2024-Q3, no progress)

## False positive (suppressed)

- internal/billing/proto.go:* — generated by protoc; ignore
- internal/billing/plugins.go:34 — `func RegisterPlugin` (registered via reflection in init)
```

### Step 4 · Confirm

Show the user the report. Wait for explicit approval. **Never delete without a "yes."**

Confirmation phrasing accepted:
- *"go ahead"*, *"delete safe + probably safe"*, *"do the safe pile only"*, *"sweep all 19"*, *"skip dead-code, just the layout move"*.

### Step 5 · Execute

For each confirmed deletion:
- One commit per category (Safe / Probably safe / Needs-human-confirmed).
- Commit message format: `chore(billing): remove unused parseRow + 11 others (anvil sweep)`
- Body lists the symbols removed with file:line citations.
- If the symbol was exported, the commit message says so loudly — even with no known consumers, it's a wire surface change.

### Step 6 · Verify

After deletion:
- Re-run the detector. New findings? Quasi-dead becomes dead once their callers are removed.
- Re-run tests. If anything breaks, revert the commit, mark the finding "needs human."
- Commit the sweep report itself to `.anvil/sweep-2026-05-22.md` for the audit trail.

## Sweep budget

Don't sweep everything at once on a mature codebase. Reasonable per-PR limits:

| Repo size | Safe-pile per PR | Probably-safe per PR |
| --- | --- | --- |
| <10k LOC | All | All |
| 10–100k LOC | ≤50 symbols | ≤20 |
| >100k LOC | ≤25 symbols | ≤10 |

Big sweeps are review-hostile and increase the chance of a false-positive landing.

## When NOT to remove

| Case | Why keep |
| --- | --- |
| **Public API in a library, even if "unused" within this module** | External consumers may import |
| **Reflection / plugin registration sites** | Tool's reachability analysis can't see them |
| **Generated code, marked as such** | Removing the marker doesn't kill the regen; just regenerates anyway |
| **Test fixtures and golden files** | Detector may flag them, they're not dead |
| **Conditional compilation per platform** | `//go:build windows` may be unused on macOS build but reachable elsewhere |
| **Feature behind a real flag, currently off but with a planned-on milestone** | Flagged-off ≠ dead if the milestone is real and dated |

For each "keep" decision, anvil prefers to add a `// nolint:unused` or `#[allow(dead_code)]` annotation with a *reason* comment over leaving the detector to re-flag it every run.

```go
// nolint:unused // reflection registration target — wired by RegisterPlugin in init()
func handleFooEvent(e Event) {}
```

## Commented-out code

Anvil's stance: **delete it. Git remembers.**

The argument for keeping commented-out blocks ("we might want it back") is almost always wrong:

- If you don't remember why it was commented, you don't remember why you'd want it back.
- `git log -S '<distinctive token>' -- path/to/file` finds it instantly.
- Commented blocks rot — function signatures around them change; the block becomes uncompilable and silently lies about being "ready to re-enable."

Exception: a 1–3 line snippet preserved as a worked example *inside* an active comment that explains *why* the surrounding code is shaped a particular way. That's documentation, not dead code.

## "TODO: remove this" hygiene

A TODO without an owner, a date, and a condition is a forever scar.

Anvil's audit flags any TODO/FIXME/XXX/HACK comment that lacks:
- A name or username (`// TODO(@george): …`)
- A condition (date, ticket, version: `// TODO: remove after v3.2`, `// TODO: remove when go 1.23 lands`)

The fix is to either delete (most often) or rewrite with both.

## Deprecated public APIs

```go
// Deprecated: Use NewClientV2. Removed in v3.0.0.
func NewClient() *Client { ... }
```

The deprecation comment is a contract with future maintainers. Anvil enforces:

- **Removal milestone declared** — version, date, or condition. `// Deprecated:` without a removal commitment is a forever scar.
- **Removed when the milestone hits** — anvil's audit cross-checks the current version against the milestone; if we've passed it, the deprecated symbol is in the sweep candidate list.
- **Loud changelog entry** when the removal happens. See `anti-patterns.md` § Wire breaks.

## Anti-patterns specific to dead code

| Tell | Why |
| --- | --- |
| **"Kept for reference"** | Git is the reference; comments rot |
| **Anonymous TODOs from years ago** | Become tombstones; no one knows what to do with them |
| **Whole files commented out** | Same as deleting + worse; the file is in the build tree but doesn't compile |
| **Unused public exports in a service binary** | Services don't have external API surface in the import sense — flag aggressively |
| **`if false { … }` debug blocks** | Use a debug flag or remove |
| **Multiple "old/", "legacy/", "deprecated/" subdirectories** | Move to docs or delete; don't carry weight |

---

## Slop gates

| Gate | Failure |
| --- | --- |
| S08 | Dead-code count above per-repo-size threshold |
| S09 | TODO/FIXME without owner + condition |
| S10 | Deprecated public API past its declared removal milestone |
| S11 | Commented-out code block > 5 lines |
| S12 | "Kept for reference" / "old impl" comments |

## Quick recipe — solo sweep

```bash
# Go
deadcode -test ./... > .anvil/dead.txt
staticcheck -checks U1000 ./... >> .anvil/dead.txt

# TS
npx knip --reporter compact > .anvil/dead.txt

# Python
vulture src/ --min-confidence 80 > .anvil/dead.txt
```

Then:

```bash
anvil reshape internal/billing --sweep
```

Anvil reads `.anvil/dead.txt` if present, otherwise re-runs the detector. Categorises, plans, asks for confirm, executes.
